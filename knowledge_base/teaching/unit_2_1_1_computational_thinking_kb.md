# Unit 2.1.1: Computational Thinking
## OCR J277 GCSE Computer Science -- Knowledge Base File

---

## Specification Requirements

**Students must know:**
- **Abstraction** means removing or ignoring unnecessary detail from a problem so that only the relevant information is kept; it allows a programmer to focus on what matters for the solution
- **Decomposition** means breaking a complex problem down into smaller, more manageable sub-problems or tasks
- **Algorithmic thinking** means defining a step-by-step logical solution that can be implemented and followed to solve a problem

**Key distinction:** When a scenario or context is given in a question, abstraction and decomposition must be applied to that specific context. Generic definitions of what the terms mean receive no marks.

**Marking language:**
- Accept: "Removing/ignoring unnecessary detail" // "Focusing on relevant information" for abstraction
- Accept: Any sensible application of abstraction to the given scenario (e.g. "storing year group instead of date of birth", "focusing on event names and ignoring student IDs")
- Accept: "Breaking the problem into smaller sub-problems/tasks/modules" for decomposition
- Accept: Any sensible application of decomposition to the given scenario (e.g. "having a separate algorithm for each event", "breaking the database into separate tables")
- Do not accept: Generic definitions of abstraction or decomposition with no reference to the given scenario
- Do not accept: Simply stating that data is split into smaller groups for decomposition -- must extend to true decomposition such as separate subroutines or separate database tables

**Not required:** Knowledge of other computational thinking concepts beyond abstraction, decomposition, and algorithmic thinking

## Examiner Intelligence

### Recurring Misconceptions (flagged in examiner reports 2022--2025)

1. **Abstraction/decomposition applied generically rather than to the scenario**
   - When a context is given (e.g. a sports day program), generic definitions gain zero marks. Abstraction must reference what detail is removed or kept in that specific context; decomposition must reference how that specific program or database is broken down.
   - Source: June 2024 J277/02 ER Q9(e) -- "No marks were given by examiners for generic definitions of what the term abstraction or decomposition means"

2. **Decomposition confused with splitting data into fields**
   - Describing how a database record is split into fields (StudentID, YearGroup etc.) is not decomposition. Decomposition means breaking the program into separate subroutines, or the database into separate tables.
   - Source: June 2024 J277/02 Q9(e)(ii) MS -- "Do not give answers discussing splitting into fields"

### Common Mark Losses
- Applying abstraction/decomposition generically rather than to the specific scenario given
- Describing field splitting as decomposition
- Confusing abstraction (removing detail) with simplification in a vague sense

### Question Patterns (2022--2025)

| Year | Paper | Question | Topic Focus | Marks | Format |
|------|-------|----------|-------------|-------|--------|
| 2022 | J277/02 | Q1(c) | Match computational thinking principles to examples | 3 | Matching |
| 2024 | J277/02 | Q9(e)(i) | Abstraction applied to sports day scenario | 1 | Identify (AO3) |
| 2024 | J277/02 | Q9(e)(ii) | Decomposition applied to sports day scenario | 1 | Identify (AO3) |

**Key observations:**
- Abstraction and decomposition applied to a specific scenario is the dominant format -- always context-driven
- Algorithmic thinking is rarely assessed as a standalone question; it underpins all algorithm design questions

## Quick Reference

| Principle | Definition | Exam trap |
|-----------|-----------|-----------|
| Abstraction | Remove/ignore unnecessary detail; focus on what is relevant | Must apply to the given scenario -- not a generic definition |
| Decomposition | Break a problem into smaller sub-problems | Splitting into fields is NOT decomposition |
| Algorithmic thinking | Define a step-by-step solution | Rarely asked standalone; underpins all 2.1.2 content |

### Key Exam Language

**Say this:**
- "Removing unnecessary detail to focus on what is relevant to the problem" (abstraction)
- "Breaking the problem into smaller, more manageable sub-problems/subroutines" (decomposition)
- Always apply to the context given -- name specific details or sub-tasks from the scenario

**Not this:**
- Generic definitions when a scenario is provided
- "Splitting the data into fields" as decomposition
